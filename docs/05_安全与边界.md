# 安全与边界

## 一、Manifest V3 约束

项目建议基于 Manifest V3 开发。

在该模式下，需要特别注意：

- 扩展代码应以打包代码为主。
- 不应依赖远程托管代码作为主要执行逻辑。
- 不建议设计为“服务端直接下发任意 JS 字符串并执行”。

## 二、推荐安全模型

### 服务端信任边界

- 仅允许连接预设白名单域名。
- 所有通信必须使用 HTTPS 或 WSS。
- 插件启动后应校验服务端地址是否合法。

### 鉴权

- 支持 API Key 或 Bearer Token。
- 认证信息存储在 `chrome.storage.local`。
- 如后续安全要求更高，可增加签名、时间戳与重放保护。

### 本地执行边界

- 仅执行插件内置能力范围内的标准化指令。
- 任意代码执行能力不应作为默认功能。
- 对危险动作应保留审计日志。

## 三、隐私保护

- DOM 上传必须区分普通页面结构和高敏感上下文字段。
- 默认不上传密码输入框的真实内容。
- 如果需求要求上传 `cookie`、`token`、`authorization` 相关值，必须视为高风险能力。
- 应允许后续增加“仅上传白名单元素”的模式。
- 不建议采集与业务无关的整页明文文本。

### 高敏感字段约束

- 应由显式开关控制是否启用。
- 仅允许发送到用户配置的可信服务端。
- 日志中不应记录这些字段的明文值。
- 如果目标是公开分发或上架，需要额外评估合规性与审核风险。

## 四、权限设计建议

推荐最小权限原则：

- `storage`
- `tabs`
- `scripting`
- `activeTab`

如确实需要固定域名范围控制，可增加受限的 `host_permissions`，避免无差别全站权限。

## 五、发布边界

如果目标是公开上架 Chrome Web Store，需要重点规避以下风险：

- 远程代码执行语义过强
- 权限范围过大且解释不足
- 数据采集范围不透明
- 缺少用户知情与审计说明

因此建议文档和实现中明确：

- 该插件执行的是标准化动作，不是通用远程脚本运行时
- 采集范围、采集时机、保存方式有清晰说明
- 服务端地址和授权方式可被用户查看和控制
