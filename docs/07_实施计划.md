# 实施计划

## 一、建议迭代顺序

### 第一阶段：搭建基础插件骨架

- 初始化 Manifest V3 项目
- 建立 `background`、`content`、`popup` 三大入口
- 完成基础配置读写

### 第二阶段：打通连接与最小闭环

- 后台连接服务端
- Popup 支持保存服务端地址、服务器联通测试和插件启动开关
- 内容脚本采集最小 DOM 快照
- 服务端可下发 `click`、`input`、`scroll` 指令

### 第三阶段：扩展浏览器能力

- 支持 `createTab`
- 支持 `closeTab`
- 支持 `switchTab`
- 回传执行结果和错误码

### 第四阶段：增强稳定性

- 增加日志
- 增加超时和重试
- 增加普通 DOM 数据与高敏感上下文字段的分级处理
- 增加连接恢复能力

## 二、建议第一期范围

第一期建议严格控制在以下范围：

- 配置服务端地址
- 支持服务器联通测试与插件启动开关
- 建立 WebSocket 连接
- 上传当前页面简化 DOM 快照
- 支持 `click`、`input`、`scroll`
- 支持 `createTab`、`closeTab`、`switchTab`
- 完成基础日志和错误回传

## 三、暂缓实现内容

- 任意 JS 字符串执行
- 复杂 DOM 动态增删改编排
- 多 iframe 深度控制
- 完整事件录制系统

## 四、主要风险

### 平台限制风险

Manifest V3 对远程执行逻辑有严格限制，需要在设计上避免踩线。

### 性能风险

整页 DOM 上传可能过大，必须做裁剪和节流。

### 选择器稳定性风险

页面结构频繁变化时，元素定位容易失效，需要设计兜底策略。

### 安全风险

如果缺少白名单、鉴权和日志，且允许上传高敏感字段，该插件容易演变成高风险远控工具。

## 五、完成定义

当以下条件全部满足时，可认为第一期完成：

- 插件可安装并运行
- Popup 可保存配置并测试连接
- 后台可与服务端稳定通信
- 内容脚本可上传 DOM 快照
- 至少 3 种页面操作和 3 种标签页操作可用
- 执行结果有明确回传与错误码
