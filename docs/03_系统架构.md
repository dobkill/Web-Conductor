# 系统架构

## 一、架构原则

- 后台统一调度，全局状态不散落在内容脚本中。
- 页面操作下沉到内容脚本，浏览器级能力保留在后台脚本。
- 配置管理、通信管理、执行引擎相互解耦。

## 二、推荐模块

### 1. Background Service Worker

职责：

- 管理插件生命周期
- 读取与维护配置
- 建立与服务端的连接
- 管理标签页与窗口操作
- 分发服务端指令到目标内容脚本
- 汇总执行结果并回传服务端

### 2. Content Script

职责：

- 采集页面 DOM 快照
- 执行页面内的元素操作
- 监听页面 DOM 事件
- 将页面内结果回传给后台脚本

### 3. Popup Page

职责：

- 配置服务端地址
- 测试连接状态
- 控制插件启动开关
- 展示接口文档与使用说明

### 4. Shared Core Modules

职责：

- 统一定义消息类型
- 定义指令模型
- 提供日志、校验、工具函数

## 三、核心数据流

### DOM 上传流程

1. 后台检测到当前页面可注入内容脚本。
2. 后台向内容脚本发起 DOM 采集请求。
3. 内容脚本生成 DOM 快照并返回后台。
4. 后台将快照发送给服务端。

### 指令执行流程

1. 服务端下发标准化指令消息。
2. 后台根据指令类型判断由谁执行。
3. 页面操作类指令转发给内容脚本。
4. 标签页操作类指令由后台直接调用 Chrome API。
5. 执行结果统一回传服务端。

## 四、职责边界

### 后台负责

- 与服务端通信
- 管理配置和连接状态
- 调用 `chrome.tabs`、`chrome.windows` 等 API
- 保存任务上下文和日志

### 内容脚本负责

- 页面 DOM 读写
- 元素定位
- 页面内事件监听
- 页面内滚动与输入

### Popup 页负责

- 用户可见配置项管理
- 可视化状态展示

## 五、为什么不建议由内容脚本直接长连接

- 内容脚本随页面刷新而销毁，连接不稳定。
- 多标签页会造成多个连接并发，增加服务端复杂度。
- 全局指令难以统一编排。

因此建议由后台脚本维护唯一主连接，再按需向内容脚本派发任务。

## 六、高敏感数据上传链路

如果需求中需要上传 `cookie`、`token`、`authorization` 相关值，建议采用独立链路处理：

1. 优先由后台脚本统一采集和发送，不与普通 DOM 快照结构完全混合。
2. 采集行为受插件启动开关和服务端配置共同约束。
3. 相关日志只记录“发生过上传”这一事实，不记录原始值。
