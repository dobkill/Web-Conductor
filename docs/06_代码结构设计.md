# 代码结构设计

## 一、目标

代码结构需要同时满足以下要求：

- 清楚区分浏览器后台逻辑与页面内逻辑
- 支持后续扩展更多指令
- 支持独立测试协议、执行器与工具模块
- 降低 Manifest V3 带来的模块耦合问题

## 三、目录职责说明

### `public/`

- 放置 `manifest.json`
- 放置图标与静态资源

### `src/background/`

- 负责插件全局调度
- 负责服务端连接和标签页控制
- 负责高敏感上下文字段的统一采集和发送

### `src/content/`

- 负责页面 DOM 读取与页面内动作执行

### `src/popup/`

- 负责 Popup 页面 UI 和交互

### `src/shared/`

- 放公共类型、常量、协议、工具和存储封装

### `src/test/`

- 放单元测试和测试夹具

## 四、模块设计建议

### 指令执行层

建议使用“路由器 + 动作处理器”模式：

- `command-router.ts` 负责根据 `action` 分发
- 各动作模块只处理单一动作
- 新增指令时不影响已有执行逻辑

### DOM 采集层

建议拆为以下职责：

- `snapshot.ts` 负责遍历与提取
- `sanitize.ts` 负责脱敏
- `selector.ts` 负责生成 XPath 和辅助定位信息

### 元素定位约定

- 服务端和插件执行层统一以 XPath 作为主定位语法。
- DOM 快照中应为关键节点生成 `xpath` 字段。
- 如需兼容人工调试，可额外保留 `selector`，但不作为主执行依据。

### 高敏感上下文采集层

建议拆为以下职责：

- `sensitive-context.ts` 负责统一收集 `cookie`、`token`、`authorization` 相关值
- 与普通 DOM 快照结构分开建模和发送

### 连接层

建议拆为以下职责：

- `ws-client.ts` 负责底层连接
- `reconnect.ts` 负责重连策略
- `session-store.ts` 负责连接态和任务态缓存

## 五、技术选型建议

- 语言：TypeScript
- 构建：Vite
- 校验：Zod 或同类 schema 校验工具
- 测试：Vitest

## 六、为什么这是合理结构

- 符合 Chrome 插件天然的多运行环境拆分
- 指令能力扩展成本低
- 协议、类型和存储逻辑可复用
- 对后续测试和维护更友好
